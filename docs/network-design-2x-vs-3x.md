# Network connection design: 2.x vs 3.x — the reliability regression investigation

**Date:** 2026-07-17
**Status:** Investigation complete; deciding hardware experiment in progress (see §7)
**Companion:** `docs/ota-check-failure-ledger.md` (the fix-attempt ledger this investigation was launched to escape; cross-referenced, not duplicated here)

---

## 1. The question, and why it took six fix attempts to ask

The owner's observation: the 2.x firmware line ran ~2 years against queue-times.com and *almost never* failed an HTTP request. From release 3.0 (2026-06-24/29) onward, outbound connections fail chronically — episodes where every NEW outbound TCP connect raises `OSError: 16` (EBUSY) at `adafruit_connection_manager._get_connected_socket` while the socket table shows 7/8 slots free, the heap is healthy, and inbound (the config web server) keeps answering. In unattended operation only a cold-radio chip reset reliably cures it — though a REPL `wifi.radio.connect()` cured it once, and the 07-17 episode flapped healthy on its own (see §5.1).

It took six logged fix attempts (ledger #0–#5, spanning the 3.5.5→3.5.18 series) to ask *why 3.x fails at all*, because every one of them built **recovery machinery around an unexamined connection path**: session rebuilds, sacrificial arenas, radio-bounce rungs, watchdog rearming, escalation counters, cold-reset budgets. Each mechanism treated the wedge as a fault to survive rather than a regression with a boundary. The reframing question — *what actually changed about how this device makes connections between 2.x and 3.x?* — had never been answered end-to-end. This document answers it.

Three boundary facts frame everything:

- Commit **08c8da5** (2026-06-24) changed the CircuitPython firmware (9.x → 10.2.1, i.e., a new ESP-IDF/lwIP/WiFi-driver generation) **and** the HTTP library stack (adafruit_requests 2.0.2 → 4.1.17, connection_manager unused → 3.1.8) **in one commit**. The themeparks.wiki data-source migration followed days later in the same release. No fielded configuration ever mixed old/new (a ~2-day dev-bench window, 2026-06-22→24, ran the rewritten app on CP 9.1.0 with the old 2.0.2 bundle during bring-up — dev-only, never fielded, too brief to bear on a days-scale onset) — so the timeline alone cannot discriminate among the candidate causes; only the physical failure signature can.
- The failing symbol, `_get_connected_socket`, **did not exist anywhere in the 2.x firmware image** (verified by `.mpy` string extraction at tag 2.5 vs main).
- The project's hardware record already contains a **warm-radio law** (ledger, "2026-07-15 evening"): a `microcontroller.reset()` issued while the station is associated carries the disease into the next session (3-for-3 both directions); a cold-radio boot is healthy; and during the ~26 h wedge, `wifi.radio.connect()` alone at the REPL cured outbound connects with every wedged object still in RAM.

## 2. Method

A 10-agent workflow (id `wf_584ead62-f2a`, 2026-07-17), three phases, ~850k tokens (845,185), 179 tool calls:

1. **Read (5 parallel readers):**
   - `net2x` — full connection lifecycle of the 2.5-era network stack (tag 2.5).
   - `net30` — the release-3.0 stack (branch release-3.0 + ScrollKit Library ~5c6ba0b/0.8.1).
   - `netnow` — the current main-branch stack (3.5.18-era app + library).
   - `bundle` — version archaeology of the vendored `.mpy` bundle and CircuitPython firmware across the repo's whole history (string extraction from compiled blobs, `git log --follow`).
   - `research` — external facts: adafruit_requests/connection_manager release history read from source, CircuitPython↔ESP-IDF version mapping verified from submodule SHAs, esp-lwip `sockets.c` EBUSY origin, upstream issue search, live payload/TLS-chain measurements of both APIs.
2. **Compare (1 comparator):** layer-by-layer design table + six ranked candidate causes with mechanisms and evidence for/against.
3. **Verify (4 adversarial refuters):** each of the top four causes handed to an independent agent instructed to *refute* it using the repo, the library, the CircuitPython 10.2.1 source, and the failure ledger.

Scope rule: recovery mechanisms (watchdog, session rebuild, radio bounce, reboot ladders) were excluded from the investigation; the warm-radio-reset observation was admitted as physical evidence only.

The adversarial phase is the heart: **all four top-ranked causes were refuted as stated.** What survives is narrower and more useful than any of them (§6).

## 4. The six ranked candidate causes

1. **Firmware-generation swap (HIGH).** The CP 9.x/ESP-IDF 5.1–5.3 → CP 10.2.1/ESP-IDF 5.5.3 move put the device on an lwIP substrate with a persistent-state defect on the outbound-connect path — most plausibly esp-lwip's `select_waiting` slot-poisoning, whose u8 overflow is the only EBUSY(16) source in `sockets.c`, with `alloc_socket` never zeroing the counter on fd-slot reuse. Field signature match: new connects fail while pooled flows and inbound keep working; reset cures.

2. **Retry-masking removal (HIGH).** requests 2.0.2's `_get_socket` caught ANY OSError from create/wrap/connect, closed the socket, freed every idle cached socket, and retried up to 5× — a transient EBUSY was invisible to the app. CM 3.1.8 retries once, and only if idle pooled sockets exist — which the current zero-idle design guarantees they never do. An identical underlying fault rate would render as "almost never fails" on 2.x and "fails constantly" on 3.x.

3. **Connect-frequency amplification (MEDIUM).** 2.x steady state minted essentially zero fresh connects (one lazily-probed keep-alive socket); current 3.x deliberately pays a full DNS+TCP+TLS connect every burst (~6/hour, ~60/hour stale), with stacked retries turning one bad park into up to 6 back-to-back attempts — converting any low-probability per-connect fault into a frequent observed event.

4. **Payload growth × timeout shrink (MEDIUM).** Per-park payload grew ~15–18× (5.5 KB → 84–92 KB) at 3.0 while the per-request timeout shrank 10× (implicit 60 s → hard 6 s); marginal-RF aborts mid-handshake/mid-body would tear down sockets mid-select (the suspected `select_waiting` leak trigger), joined by MemoryError teardowns from internal-SRAM pressure (~40 KB mbedTLS context per pooled TLS socket in the ~320 KB PSRAM-unreachable SRAM).

5. **Global-pool occupancy (MEDIUM).** At 3.0, CM kept freed sockets OPEN across 300 s gaps, ≥4 SocketPools existed per boot, and the OutOfRetries recovery orphaned whole pools+sockets forever in CM's module-global registry (no `close_all` existed) — pushing the small lwIP socket table toward the near-max regime where esp-lwip's documented select races (esp-lwip#13) live, and occasionally to outright exhaustion (`RuntimeError: Out of sockets` in the ledger). Largely mitigated in 3.5.x, yet failures continue — an aggravator of the 3.0–3.5 window, not sufficient for today's wedges.

6. **Modem power-save × mesh × new radio stack (LOW).** Power-save default ON in both eras, but the doze/wake and roam implementation changed across IDF generations; frames lost around the SYN/ClientHello window would fail fresh connects preferentially, and a mid-select radio transition is itself a candidate leak trigger. Under live A/B test since 2026-07-16 (`power_management=NONE`); no result yet, and the setting may not survive an in-firmware roam.

## 5. The refutations

Four adversarial agents each tried to kill one of the top four causes using the repo, the CircuitPython 10.2.1 source, and the failure ledger. **All four verdicts: REFUTED** — each on the project's own hardware evidence, not on argument. This section is the document's permanent value: these mechanisms must not be re-proposed without new evidence.

### 5.1 lwIP `select_waiting` slot-poisoning — REFUTED

The claim's load-bearing feature — poison persisting in lwIP's static `sockets[]` array until "chip reset zeroes it" — is contradicted three independent ways by the ledger:

1. **Cure without reset.** During the ~26 h wedge (socket table 7/8 free), `wifi.radio.connect()` alone at the REPL restored outbound TCP+TLS with every wedged object still in RAM (ledger, "2026-07-15: the evidence session"). Reassociation cannot clear a `select_waiting` counter, and the next connect would reuse the same lowest-free slot — yet it succeeded.
2. **Disease through a chip reset.** The warm-radio law (3-for-3 both ways): `microcontroller.reset()` while associated carries the failure into the next session — whose `.bss`, *including `sockets[]`*, is freshly zeroed at boot — while a cold-radio boot with identical software is healthy. The state variable provably lives in the radio/PHY domain (survives CPU reset, cleared by radio power-down), not in lwIP's structures. The claim predicts warm reset cures as well as cold; the field shows the opposite.
3. **Flapping.** The wedge flaps sick↔healthy on a minutes scale with no reboot, and arose in-session from a POWER_ON boot in <1 h (ledger, "2026-07-17: the flapping falsification"). A counter stuck at 255 is arithmetically sticky and cannot flap.

The mechanism is also internally inconsistent with the code it quotes: EBUSY fires only on the 255→0 wrap, so a single leaked increment causes nothing — poisoning needs ~255 accumulated leaks per slot (implausible at this app's connect rate) or a decrement-underflow via mid-select slot reuse, the exact race lwIP ≥2.1 `fd_used` refcounting plus the savannah-57445 fix (closed 2020) is believed to prevent — though whether that fix is present in the exact esp-lwip submodule vendored by CP 10.2.1 was NOT verified (UNCONFIRMED; worth a one-time diff).

**What survives:** verified in CP 10.2.1 `Socket.c`, only `connect()` uses `lwip_select` (recv/accept are nonblocking retry loops; zero EBUSY anywhere in `Socket.c` — the errno is manufactured in esp-lwip's `sockets.c` and reaches Python only through `connect()`'s `lwip_select` loop) — so `lwip_select`'s overflow check remains the most plausible *proximate emitter* of errno 16, elegantly matching "outbound connects die, inbound fine" — but as a symptom re-created by warm-radio-state-dependent behavior each session, not a persistent substrate defect. The generic "CP10 changed the substrate" correlation stands; the persistence model is falsified.

**Decisive experiment:** at the next live wedge, at the REPL without rebooting: allocate two fresh sockets first (occupying the lowest free lwIP fd slots), then attempt the failing connect on a third. Slot-poisoning predicts displacement off the poisoned slot makes the connect succeed; radio-state predicts EBUSY on every slot regardless.

### 5.2 The lost 5× connect-retry masking — REFUTED

The code delta is real (source-verified: requests 2.0.2/2.0.5 `_get_socket` 5× catch-all close/free/retry loop at lines 471–530 vs CM 3.1.8 `get_socket`'s single retry gated on `available_socket_count > 0`; `_get_connected_socket` exists only in the 3.x `.mpy`). The causal claim fails on three contradictions:

1. **Retries cannot mask a persistent fault.** The 2026-07-15 USB probe of a wedged box: EVERY new outbound connect failing `OSError: 16` *for hours*, 7/8 slots free, curable only by `wifi.radio.connect()` — an operation requests 2.0.2's loop never performs. Its 5 close-and-reconnect attempts would all fail identically, then OutOfRetries → the 2.x HttpClient's pool+session rebuild (also not a radio reconnect) → same wedge. An identical fault in the 2.x era would have wedged 2.x visibly; 2.x's observed reliability therefore cannot be retry masking — **the fault must not have occurred on the CP 9.x-era stack.**
2. **The zero-idle premise is anachronistic.** `close_pooled_sockets` landed in 3.5.18 (commit b6eda42, 2026-07-16/17), AFTER the entire chronic EBUSY history — explicitly so "no surviving socket can mask the wedge." Through 3.5.17, pooled keep-alive sockets idled across gaps and kept working; new-connect EBUSYs occurred WITH idle sockets present — the exact condition arming CM's free-all-and-retry branch — and the failures were chronic anyway.
3. **3.x retries more, in wall-clock terms, and still fails.** 2 (service) × 3 (HttpClient) fresh connect attempts per park plus a 60 s stale cadence give more retry coverage than 2.0.2's sub-second 5× loop — yet the 07-17 session shows phase-correlated flapping (all attempts fail in sick phases, all succeed in healthy ones), incompatible with the era-invariant independent fault rate the claim requires.

**What survives:** at most, the retry removal marginally sharpens visibility of rare genuine one-shot transients (see §6).

**Decisive experiment:** on the next wedged specimen (do NOT power-cycle), replicate 2.0.2's semantics at the REPL: loop 5× { `pool.socket()` → wrap → connect to api.themeparks.wiki:443; on OSError close and retry }. Prediction: all 5 raise Errno 16; only `wifi.radio.connect()` clears it. Alternatively, grep past serial logs for ≥5 consecutive Errno-16 connect attempts within one burst — that settles it without hardware.

### 5.3 Connect-frequency amplification — REFUTED

1. **Temporal inversion.** The elements creating today's high fresh-connect rate — the zero-idle drain and per-burst "honest probe" (main:src/app.py:834, "owner's design rule 2026-07-16") and the 600 s cadence — shipped in 3.5.18. But the chronic wedge was fully established 2026-06-30→07-11 (ledger attempts 0–4) under the release-3.0 design, which held keep-alive across the gap exactly like 2.x (`_close_response` returns the socket to CM's available-pool, release-3.0:src/api/theme_park_service.py:17,137,146; no `connection_manager_close_all` existed at library ref 5c6ba0b) at the same 300 s cadence. **Connect frequency at onset was the same order as 2.x** — the cause must live elsewhere.
2. **False baseline.** "2.x ≈ zero fresh connects" is unproven and likely false: whenever the CDN dropped the idle keep-alive (idle window unknown), 2.0.2's liveness probe minted a fresh DNS+TCP+TLS connect per burst — up to ~12/hour, MORE than current 3.x's ~6/hour healthy and only 5–10× below stale mode. Nowhere near "orders of magnitude."
3. **Arithmetic + signature.** The 2.x stack accumulated thousands of fresh-connect draws over ~2 years (boots, WiFi drops, dead keep-alives, and a retry fan-out DEEPER than 3.x's — up to ~270 connect attempts for a bad boot catalog vs 3.x's 27) with zero chronic wedges, while 3.x wedges within days. And the wedge is a persistent poisoned *state*, not a per-draw Bernoulli event: multiplying draws shortens time-to-detection but cannot manufacture the reliable-vs-chronic contrast unless the per-connect fault probability itself changed with the stack.

**What survives:** connect frequency (especially the 60 s stale mode, present since 3.0) is a minor observation-rate amplifier and detection aid — nothing more.

**Decisive experiment:** run one soak device in a 2.x-equivalent connect profile on the current 3.x stack (disable `close_pooled_sockets()`, reuse the keep-alive socket until failure) against a stock-3.5.18 companion. If the low-connect-rate device still wedges on the same multi-day timescale (as it already did under the identical keep-alive profile of 3.0–3.5.5, per the ledger), frequency is dead as a cause.

### 5.4 Payload growth × 6 s timeout — REFUTED

The premises are real (payload ~15–18×, measured; `timeout=6` introduced at 3.0 in the library's `http_client.py:95–131` with the watchdog invariant). The mechanism fails at every joint:

1. **Per-operation timeout semantics.** The 6 s timeout is a per-socket-OPERATION deadline, not per-request wall clock: CP 10.2.1's `Socket.c` starts a fresh tick count per `recv_into` call. A slow 90 KB transfer does NOT abort for being slow — only a single ≥6 s dead-air stall aborts. The project's own logs prove it: the ledger's 3.5.16 entry records synchronous requests blocking the loop 10–19 s *and completing*. The "shrunken timeout aborts long transfers" coupling is false in the common case.
2. **The extra on-socket time never touches the select machinery.** Verified in CP 10.2.1 `Socket.c`: only `connect()` runs the `lwip_select` 100 ms polling loop (the one in-tree EBUSY source); recv is nonblocking `lwip_recv` with no select, send is bare `lwip_send`. The 15–18× body growth adds time in code that spends ZERO time in select loops. Select exposure scales with connect count — which did not grow at 3.0 (§5.3).
3. **A timeout abort cannot tear down a socket "mid-select."** The VM is single-threaded: when a timeout fires, the polling loop has already returned before the exception propagates and the close executes — a strictly sequential, ordinary close. The only actor that can destroy a socket while a select is in flight is the lwIP/wifi task during a radio disconnect/reset — precisely the independently verified warm-radio trigger, whose controlled experiment held payload and timeout CONSTANT and flipped only radio state.
4. **The episodes contradict accumulation-and-persistence.** Wedge with 7/8 slots free and healthy heap; cured by bare `wifi.radio.connect()`; flaps on a minutes scale; onset in <1 h from a cold POWER_ON boot (≤~6 refresh bursts — far too few fetches to accumulate hundreds of leaked select increments, and no preceding timeout storm recorded).
5. **The SRAM/MemoryError leg is orthogonal.** Heap was healthy during EBUSY episodes; the `-16256` PK_ALLOC failures were deterministically root-caused to raw.githubusercontent.com's RSA-2048 chain (fresh-boot reproducible; park fetches "never fail a handshake") and fixed via the ECDSA check host.

**What survives:** bigger bodies + a 6 s per-op timeout modestly raise the odds of an occasional aborted request on bad RF — a visibility effect (via §6's retry story), not a wedge mechanism.

**Decisive experiment:** point a test build at a local throttling proxy that stalls >6 s mid-body, forcing ~50 consecutive timeout-aborted fetches on a cold-booted device, then probe fresh-connect health with the existing errno-16 strike instrumentation. If 50 forced aborts never wedge while one warm `microcontroller.reset()` still does (already 3-for-3), the claim is dead. Zero-cost pre-check: grep soak logs for ETIMEDOUT preceding any errno-16 onset — the 07-17 episode already shows onset without one.

## 6. What survives: the conclusion

Every *app-level design change* proposed as the origin of the regression is refuted. What remains standing:

1. **The regression boarded with commit 08c8da5 (2026-06-24) — and the physical signature points at the firmware generation, not the Python stack.** The 2.x app shape (always-on :80 listener + periodic HTTPS to one host) ran ~2 years on pre-IDF-5.4 lwIP without this failure mode; every fielded 3.x build (per the release record) runs CP 10.2.1/ESP-IDF 5.5.3 and wedges within days-to-hours. One caveat: a library docstring records testing “on both CP 9.2.7 and the target CP 10.2.1” around the 3.0 release — if a CP 9.2.7 window existed on-device, it slightly blurs the firmware boundary (see §8).
2. **The defective state lives in the WiFi driver/radio layer.** The warm-radio law is the single most discriminating fact in the record: the disease crosses a `microcontroller.reset()` (which zeroes all of lwIP's RAM structures) yet is cured by reassociation or a cold-radio boot. That locates the state variable below lwIP's data structures, in the ESP32 WiFi driver/PHY domain that persists across a warm CPU reset. `lwip_select`'s EBUSY path in `connect()` remains the plausible *messenger* — errno 16 is manufactured there each session — but the message's cause is radio-session state, re-created rather than accumulated.
3. **The requests 2.0.2 5×-retry removal explains the *feel*, not the disease.** 2.x's silent close/free-all/retry loop made the old stack look flawless against genuine one-shot transients; 4.1.17 + CM 3.1.8 surfaces the first connect error of every zero-idle burst raw. That is a real visibility delta worth remembering (and possibly worth reinstating as a bounded connect-retry in HttpClient) — but it cannot mask a fault where every connect fails for hours, so it cannot be the regression.
4. **The confound is real and only an experiment can break it.** 08c8da5 changed firmware and library together; no fielded configuration ever mixed them. Hence §7.

## 7. The deciding experiment: CircuitPython 9.1.0 + unmodified 3.5.19

Started 2026-07-17 evening (ledger, "THE FIRMWARE EXPERIMENT"). Protocol: the desk box was flashed to CP 9.1.0 (2024-07-10 build) with UNMODIFIED app 3.5.19 — filesystem survived the downgrade; the `power_management` API is absent on 9.1, so power-save runs at the 2.x-era default ON (logged at boot); same room, same two-AP mesh, same parks; the `.mpy` bundle loads fine (format stable across 9/10). First boot: 140-park catalog + 4/4 parks fetched on first attempts.

**Baseline to beat:** on CP 10.2.1 the wedge struck multiple times per day with 20–40 min session onsets (the strike ledger logs every one with `t=`/`bssid`).

**Verdict criteria:** 24 h with zero strikes on 9.1.0 = firmware generation convicted (fix path: track/bisect CP releases + upstream reproducer). Wedge recurs on 9.1.0 = firmware exonerated; the driver-state hunt continues.

**First status update — the instrument broke.** On 9.1.0 the check button dies with `RuntimeError: pystack exhausted` (the deep handler→OTA→requests-4.x call chain overflows the 2024 firmware's default Python call-stack BEFORE any network I/O — button clicks probe nothing), and park refreshes died with MemoryError (16×) until the GENERIC 12-strike backstop auto-rebooted. Critically, the 6-strike errno-16 wedge ledger correctly refused to classify pystack/MemoryError as wedge evidence — **zero false strikes; classifier integrity held in the field.** But sessions never reach the 20–40 min onset window, so "no wedge on 9.1.0" is currently *unfalsifiable*. This is a resource misfit of a 2026 app on 2024 firmware defaults (consistent with the much-smaller 2.x app fitting 9.x comfortably), not a data point about the wedge. Repair options (owner to choose, one at a time): (a) `CIRCUITPY_PYSTACK_SIZE` bump in `/settings.toml` (one line, REPL-writable — mind the 60 s armed watchdog at the REPL), retest, see whether the MemoryError persists; (b) if the full app won't fit: a minimal reproducer script (always-on listener + timed fresh HTTPS connects, a few KB) run identically on 9.1.0 and 10.2.1 — cleaner science and the upstream-issue artifact anyway. CP 9.2.x is the fallback old-generation testbed.
